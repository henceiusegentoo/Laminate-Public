== 2. Architecture Overview
// ============================================================

=== 2.1. What Laminate adds

Laminate does not replace Paper's patch stack.
It layers a narrow, deep capability on top of upstream Paper:
loading plugins that can transform live server bytecode via
https://github.com/SpongePowered/Mixin[SpongePowered Mixin].

The central problem is ordering.
The JVM instrument API (<<transformer,`ClassFileTransformer`>>) must be active before a
class is first loaded; certain Mixin operations (field addition) cannot be applied
retroactively.
Paper's plugin system loads classes far too late.
Laminate therefore pushes the entire mixin setup to the very beginning of the JVM's life,
before any Minecraft class is loaded.

=== 2.4. Key design invariants

* The `ClassFileTransformer` must be installed *before* any `net.minecraft.*` class is
  loaded. `PaperBootstrap.boot()` calls bootstrap before `SharedConstants.tryDetectVersion()`.
* Bootstrap is _idempotent_: a second call returns cached results immediately.
* Failed hybrid plugins are _isolated_: their providers are removed from Paper's provider
  storage; the rest of the server continues starting.
* Mixin transformations are _irreversible process state_: hot-reload is blocked whenever
  any hybrid plugin is active.
